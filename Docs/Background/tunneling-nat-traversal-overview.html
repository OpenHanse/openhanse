1. The Problem: NAT & Mobile Networks

When an iPhone on 5G needs to talk directly to a MacBook on a local LAN, both devices sit behind NATs — the iPhone behind Carrier-Grade NAT (CGNAT), and the MacBook behind a home/office router NAT. Neither device has a publicly routable IP that the other can reach.

Types of NAT

  • Full Cone — easiest to traverse, any external host can reach the mapped port
  • Address-Restricted Cone — only hosts the internal has contacted can reply
  • Port-Restricted Cone — only the exact host:port pair can reply
  • Symmetric NAT — different mapping per destination; hardest to traverse (common on mobile/5G CGNAT)

Why 5G / Mobile Makes It Harder

  • Carriers deploy Symmetric NAT or CGNAT extensively
  • STUN-based hole punching often fails (~10-15% of WebRTC calls need TURN relay)
  • IP addresses change frequently during handovers
  • Some carriers block or throttle UDP traffic
  • IPv6 can help bypass NAT entirely, but dual-stack support varies

2. NAT Traversal Fundamentals

All modern P2P systems use variations of three core protocols, often combined via ICE:

STUN Session Traversal Utilities for NAT

Lightweight protocol where a client contacts a public STUN server to discover its own public IP:port mapping. Both peers exchange these "reflexive candidates" and attempt direct UDP hole-punching.

Zero relay cost — fully P2P once connected
Fails with Symmetric NAT (common on 5G CGNAT)

TURN Traversal Using Relays around NAT

Fallback relay server that forwards all traffic when direct connection fails. Both peers send data to the TURN server which relays it. Guarantees connectivity but adds latency and bandwidth cost.

Works through any NAT type, even Symmetric
All traffic relayed — adds latency + server cost

ICE Interactive Connectivity Establishment

Framework that orchestrates STUN and TURN. Gathers all candidate paths (host, server-reflexive, relay), tests connectivity in priority order, and picks the best working path. Used by WebRTC, Tailscale, and others.

Always finds a connection — graceful fallback
Initial connection setup takes 1-3 seconds

Birthday Attack / Port Prediction

Advanced technique for Symmetric NAT: both peers send many packets to predicted port ranges simultaneously, hoping for a match. Tailscale and others use this to achieve ~94% direct connection rates even through CGNAT.

Can punch through Symmetric NAT without relay
Probabilistic — not guaranteed; needs coordination

Key Insight: UDP is King for NAT Traversal

TCP-based protocols (SSH, TLS tunnels, SOCKS) fundamentally cannot do NAT hole-punching because TCP's three-way handshake doesn't survive the timing requirements of simultaneous open. All successful P2P NAT traversal is UDP-based — this is why WireGuard, QUIC, and WebRTC are the relevant protocols for your use case.

QUIC / MASQUE — The Emerging Standard

QUIC deserves special attention: by 2026, HTTP/3 (built on QUIC) handles 40-60% of web traffic. QUIC's connection migration means sessions survive IP changes (critical for mobile handovers). The MASQUE protocol (RFC 9298/9484) builds on QUIC to provide standardized UDP and TCP proxying over HTTP/3, and Cloudflare is already deploying it in their WARP product. For a Swiss company, this is particularly relevant as MASQUE is becoming the IETF standard approach to tunneling.

Detailed Breakdown

Top Pick Tailscale / Headscale

Built on WireGuard with the most sophisticated NAT traversal engine. Tailscale's coordination server (or self-hosted Headscale) distributes peer keys and endpoint info. Their NAT traversal uses an enhanced ICE-like algorithm with "birthday attack" port prediction that achieves direct connections ~94% of the time — even through Symmetric NAT / CGNAT.

  • Kernel-mode WireGuard on Linux = near-native throughput
  • DERP relay servers as fallback (encrypted, Tailscale-operated or self-hosted)
  • MagicDNS for human-friendly device names
  • ACLs, Taildrop file sharing, SSH, Funnel (public ingress)
  • Native iOS & macOS apps with seamless reconnect

ZeroTier

L2 overlay — acts like a virtual Ethernet switch. Devices get virtual MAC addresses and can broadcast/multicast. More flexible than L3 (supports non-IP protocols) but slightly more overhead.

  • Custom cryptographic protocol (not WireGuard)
  • Planetary root servers assist NAT traversal
  • Good for multi-protocol scenarios (e.g., mDNS, Bonjour)
  • BSL license: free for personal use, license for commercial

Nebula (by Slack/Defined Networking)

Fully open-source overlay with a strong security model based on certificate-based identity. Requires running your own "lighthouse" node for peer discovery, giving you full control but more setup effort.

  • MIT licensed — no vendor lock-in
  • Certificate-based auth (not tied to any identity provider)
  • Good NAT traversal but less sophisticated than Tailscale
  • Mobile support via community apps

NetBird

Newer WireGuard-based mesh that uses standard ICE (STUN/TURN) for NAT traversal. Fully open-source with a clean management UI. Good middle ground between Tailscale's ease and Nebula's self-hostability.

  • BSD-3 license, fully self-hostable
  • Standard ICE/STUN/TURN — well-understood NAT traversal
  • Built-in access control and SSO integration
  • Growing ecosystem and active development

What Happens in Practice

With Swiss mobile carriers (Swisscom, Sunrise, Salt), most deploy CGNAT with either Port-Restricted or Symmetric NAT. Tailscale's aggressive port prediction typically achieves a direct connection within 2-5 seconds. When the iPhone switches from 5G to WiFi (or vice versa), WireGuard's stateless nature means the tunnel re-establishes in under a second — the MacBook-side connection stays stable.

What You Can Do Over This Tunnel

Once the WireGuard tunnel is up, you have a full L3 IP path between the devices. This means SSH, HTTP, file sharing, screen sharing, database access — anything TCP or UDP. Tailscale even includes Taildrop for drag-and-drop file transfer between devices.

7. Recommendation

Primary: Tailscale (or Headscale for sovereignty)

For your iPhone 5G ↔ MacBook scenario, Tailscale is the clear winner. It combines the best NAT traversal engine in the industry with WireGuard's performance, native iOS/macOS apps, and zero-config setup. For a Swiss company concerned about data sovereignty, Headscale (the open-source coordination server) lets you self-host the control plane while keeping all the NAT traversal benefits.

Quick Win Tailscale Free/Personal

Get started in 5 minutes. Install on iPhone + MacBook, sign in with SSO, devices connect automatically. Free for up to 100 devices / 3 users.

Production Headscale + Tailscale Clients

Self-host Headscale on your infrastructure (Swiss datacenter). Use official Tailscale clients on all devices. Full sovereignty over coordination data, keys never leave your infra. Deploy DERP relay in Zurich for lowest-latency fallback.

Alternative NetBird (if full OSS is a must)

If you want a fully open-source stack including the management server, NetBird is the strongest alternative. Standard ICE/STUN/TURN, WireGuard data plane, self-hostable, good iOS support.

When NOT to Use These

SSH Tunneling

Still useful for quick ad-hoc port forwards to servers you already have SSH access to. Not suitable for P2P between two NATed devices.

Raw WireGuard

Great if one endpoint has a public IP (e.g., your cloud server). Missing the coordination and NAT traversal layer needed for the mobile-to-LAN scenario.

WebRTC

Best for browser-based real-time communication (video, audio, data channels). Overkill and heavyweight if you just need a network tunnel.

QUIC / MASQUE

Watch this space — MASQUE is becoming the IETF standard for tunneling and Cloudflare is leading adoption. Not yet practical as a standalone P2P mesh solution, but likely the future transport layer for next-gen overlay networks.
