1. Our Scenario

We have two devices that need to communicate with each other:

Device A — iPhone

  • Connected via 5G mobile network
  • Private IP assigned by carrier: 10.48.12.97
  • Sits behind the carrier's NAT (CGNAT)
  • Running a service (e.g. a small HTTP server) on port 8080

Device B — MacBook

  • Connected via office/home LAN
  • Private IP assigned by router: 192.168.1.42
  • Sits behind the office/home NAT router
  • Running a service (e.g. SSH server) on port 22

Goal: The iPhone wants to connect to the MacBook's SSH server, and the MacBook wants to reach the iPhone's HTTP server. Neither can.

2. IP Addresses: Public vs. Private

Every device on the Internet needs an IP address to send and receive data. But there aren't enough IPv4 addresses for every device (only ~4.3 billion exist). The solution: private IP addresses that are reused inside local networks, combined with a public IP address shared by the router that connects to the Internet.

Key point: Your iPhone's address 10.48.12.97 and your MacBook's address 192.168.1.42 are both private. No device on the Internet can send a packet to these addresses — they simply don't exist outside their respective local networks.

3. What is NAT and Why Does It Exist?

Network Address Translation (NAT) is the mechanism that allows multiple devices with private IPs to share a single public IP address. Your router performs NAT: it rewrites the source address of outgoing packets from the private IP to its own public IP, and remembers the mapping so it can route replies back to the correct device.

── How NAT works for an OUTGOING connection ── iPhone (private) Carrier NAT Internet Web Server 10.48.12.97:5000 ──▶ Rewrites to: ──▶ Packet arrives 93.184.216.34:443 85.195.22.1:61234 from 85.195.22.1 NAT Table entry: 10.48.12.97:5000 ↔ 85.195.22.1:61234 ↔ 93.184.216.34:443 Reply comes back: 93.184.216.34:443 ──▶ Looks up table: ──▶ Forwards to: port 61234 → 10.48.12.97:5000 10.48.12.97:5000 ✓ Works!

NAT is transparent for outgoing connections: the iPhone opens a connection to a web server, the NAT creates a mapping, and replies flow back through that mapping. The device behind NAT doesn't even know it's happening.

4. The Core Problem: Incoming Connections

4.1 Why Outgoing Connections Work

When your iPhone opens a connection to google.com, the NAT creates a mapping entry in its table. Google's reply arrives at the NAT's public IP and port, the NAT looks up the mapping, and forwards the reply to the iPhone. The connection works because the iPhone initiated it, which created the mapping.

4.2 Why Incoming Connections Fail

Now imagine the MacBook wants to connect to the iPhone's HTTP server. The MacBook would need to send a packet to the iPhone — but to which address?

MacBook MacBook's Router Internet Carrier NAT iPhone 192.168.1.42 ──▶ 85.195.44.8 ──▶ ??? ??? 10.48.12.97:8080 Problem: What destination IP does the MacBook use? Option A: 10.48.12.97 ──▶ ✗ FAILS — This is a private IP. The MacBook's router doesn't know where to route it. It's not on the MacBook's local network. Option B: 85.195.22.1 ──▶ Reaches the carrier NAT, but... (carrier's IP) The NAT has no mapping entry for this incoming packet. The iPhone never sent anything to the MacBook, so no NAT table entry exists. ──▶ ✗ PACKET DROPPED
This is the fundamental problem: A NAT only forwards incoming packets if they match an existing mapping — and mappings are only created by outgoing connections from the device behind the NAT. If nobody inside has talked to you first, your packets are silently dropped.

4.3 The Double-NAT Deadlock

In our scenario, both devices sit behind a NAT. Neither can initiate a connection to the other:

┌──────────────┐ ┌──────────────┐ │ iPhone │ Carrier NAT Office NAT │ MacBook │ │ 10.48.12.97 │──▶ 85.195.22.1:? ??? 85.195.44.8:? ◀──│ 192.168.1.42 │ │ │ │ │ │ "I want to │ I don't know the I don't know the │ "I want to │ │ reach the │ MacBook's private iPhone's private │ reach the │ │ MacBook" │ IP or NAT port IP or NAT port │ iPhone" │ └──────────────┘ └──────────────┘ DEADLOCK: Each device needs the other to send first, but neither can. • The iPhone can't reach 192.168.1.42 — that IP doesn't exist on the Internet • The MacBook can't reach 10.48.12.97 — that IP doesn't exist on the Internet • Even if they know each other's NAT public IPs, the NATs have no mapping entries to forward the packets to the correct internal device
Analogy: Think of NAT like a hotel reception desk. If a guest (your device) calls an outside number (outgoing connection), the receptionist notes the room number so they can transfer the callback. But if someone from outside calls the hotel and asks for "room 42" without a prior call log, the receptionist doesn't know who they are and won't connect them. Now imagine both people are in different hotels — neither receptionist will connect the call.

5. CGNAT: The Mobile Network Problem

Mobile carriers face an even bigger address shortage. They use Carrier-Grade NAT (CGNAT) — a second layer of NAT at the carrier level. Your iPhone might go through two NATs before reaching the Internet:

iPhone Carrier Internal NAT Carrier Public NAT Internet 10.48.12.97 ──▶ 100.64.5.18 ──▶ 85.195.22.1 ──▶ ... private CGNAT range (shared) public (shared by thousands of users) The 100.64.0.0/10 range is reserved specifically for CGNAT. You cannot get a port forwarding rule on your carrier's NAT. The public IP 85.195.22.1 is shared with thousands of other subscribers.

With a home router, you could at least configure port forwarding (a manual NAT mapping) to expose a service. With CGNAT, that option is completely gone — you don't control the carrier's NAT equipment. This makes mobile devices especially hard to reach from the outside.

6. How Solutions Overcome NAT

Since neither device can directly reach the other, all solutions rely on the same core idea: use a third party that both devices CAN reach — a server with a public IP address on the Internet. The solutions differ in how much they rely on that server.

6.1 Traditional VPN — Relay Through a Server

The simplest approach: both devices connect outward to a VPN server in the cloud. Since outgoing connections work through NAT, both can establish a tunnel to the server. The server then routes traffic between them.

┌──────────────┐ ┌──────────────────┐ ┌──────────────┐ │ iPhone │ outgoing │ VPN Server │ outgoing │ MacBook │ │ 10.48.12.97 │────tunnel──────▶│ 203.0.113.50 │◀──────tunnel────│ 192.168.1.42 │ │ │ (works!) │ (public IP) │ (works!) │ │ │ VPN IP: │ │ │ │ VPN IP: │ │ 10.8.0.2 │◀───encrypted───▶│ Routes between │◀───encrypted───▶│ 10.8.0.3 │ └──────────────┘ traffic │ VPN clients │ traffic └──────────────┘ └──────────────────┘ ✓ Both connections are OUTGOING → both pass through NAT successfully The VPN server acts as a relay, forwarding packets between the two clients. Each device gets a virtual private IP (10.8.0.x) on the VPN network.
1

iPhone opens an outgoing encrypted tunnel to VPN server 203.0.113.50 → Carrier NAT creates a mapping → connection established.

2

MacBook opens an outgoing encrypted tunnel to the same VPN server → Office NAT creates a mapping → connection established.

3

The VPN server assigns virtual IPs (e.g. 10.8.0.2 for iPhone, 10.8.0.3 for MacBook).

4

iPhone sends a packet to 10.8.0.3 (MacBook's VPN IP). The packet travels through the tunnel to the VPN server, which forwards it through the MacBook's tunnel.

Downside: All traffic between the two devices passes through the VPN server, even if the devices are on the same WiFi network. This adds latency (especially if the server is geographically distant) and the server's bandwidth becomes a bottleneck.

Examples of this approach: OpenVPN, WireGuard (in client-server mode), IPsec/IKEv2.

6.2 Overlay Networks — The Smart Approach (ZeroTier, Tailscale)

Overlay networks like ZeroTier and Tailscale improve on the basic VPN by adding intelligence: they use a coordination server for setup but then attempt to establish a direct peer-to-peer connection between the devices. If direct connection isn't possible, they fall back to relaying.

── Phase 1: Discovery (via coordination server) ── ┌──────────────┐ ┌──────────────────┐ ┌──────────────┐ │ iPhone │ outgoing │ Coordination │ outgoing │ MacBook │ │ │───registers────▶│ Server │◀───registers────│ │ │ │ │ │ │ │ │ │◀───receives ────│ "MacBook is at │────receives───▶ │ │ │ │ peer info │ 85.195.44.8 │ peer info │ │ │ │ │ port 51820" │ │ │ └──────────────┘ └──────────────────┘ └──────────────┘ ── Phase 2: Direct connection attempt (NAT hole punching) ── ┌──────────────┐ ┌──────────────┐ │ iPhone │──────── UDP packet to 85.195.44.8:51820 ──────────▶ │ MacBook │ │ │◀────── UDP packet to 85.195.22.1:43210 ──────────│ │ │ │ │ │ │ │ Direct P2P tunnel established! │ │ │ │◀═══════ encrypted traffic flows directly ══════════▶│ │ └──────────────┘ (no server in the middle) └──────────────┘ ── Fallback: If direct connection fails ── ┌──────────────┐ ┌──────────────────┐ ┌──────────────┐ │ iPhone │═══encrypted════▶│ Relay Server │◀═══encrypted═══│ MacBook │ │ │◀══traffic═══════│ (DERP / Root) │════traffic════▶ │ │ └──────────────┘ └──────────────────┘ └──────────────┘

Here's how ZeroTier specifically handles it:

1

Registration: Both devices install the ZeroTier client and join the same virtual network (identified by a 16-digit network ID). Each client connects outward to ZeroTier's "Planet" root servers — this works through NAT because it's an outgoing connection.

2

Peer discovery: The root server tells each device about the other's public IP and port (as seen from the Internet side of the NAT). Think of this like a mutual friend introducing two people.

3

Hole punching attempt: Both devices simultaneously send UDP packets to each other's discovered public IP:port. When a NAT sees an outgoing packet to a specific address, it creates a mapping — and if the other device's packet arrives at that mapping in time, the NAT allows it through. This is called UDP hole punching (explained in detail in section 6.3).

4

Direct tunnel or relay: If hole punching succeeds (~90% of the time), all traffic flows directly between the devices — the root server is no longer involved. If it fails (due to very restrictive NAT), traffic relays through ZeroTier's root servers as a fallback.

Key advantage: The coordination server only handles tiny control messages (key exchange, endpoint info). Actual data flows peer-to-peer in most cases. This means low latency and no bandwidth bottleneck on the server — unlike a traditional VPN where all traffic goes through the server.

6.3 UDP Hole Punching — How Direct P2P Works

The trick that makes direct connections possible is called UDP hole punching. It exploits how NAT creates mappings for outgoing packets:

── Setup: Coordination server knows both peers' public endpoints ── iPhone NAT mapping: (none yet) MacBook NAT mapping: (none yet) ── Step 1: Coordination server tells both peers to send simultaneously ── iPhone sends UDP to 85.195.44.8:51820 (MacBook's NAT public side) → Carrier NAT creates mapping: 10.48.12.97:5000 ↔ 85.195.22.1:43210 → 85.195.44.8:51820 → Packet arrives at MacBook's NAT... → MacBook's NAT has no mapping yet → packet dropped (but that's OK!) MacBook sends UDP to 85.195.22.1:43210 (iPhone's NAT public side) → Office NAT creates mapping: 192.168.1.42:51820 ↔ 85.195.44.8:51820 → 85.195.22.1:43210 → Packet arrives at iPhone's Carrier NAT... → Carrier NAT HAS a mapping for 85.195.44.8:51820 → ✓ PACKET FORWARDED to iPhone! ── Step 2: Now the "hole" is punched in BOTH NATs ── iPhone sends another UDP to MacBook → Arrives at MacBook's NAT, which now has a mapping for iPhone's address → ✓ FORWARDED to MacBook! Both NATs now have active mappings → bidirectional communication established!
Why UDP and not TCP? UDP is connectionless — you can send a packet without a handshake. TCP requires a three-way handshake (SYN → SYN-ACK → ACK) which must complete in order. Simultaneous TCP opens are theoretically possible but fail in practice because most NATs expect to see the handshake in the correct sequence. UDP packets can simply "fly" in both directions and the NATs will create mappings for them regardless.

Hole punching doesn't always work — particularly with Symmetric NAT (common on mobile carriers), where the NAT assigns a different external port for each destination. Advanced techniques like port prediction and the "birthday attack" (sending to many ports simultaneously) can overcome this in many cases, which is why Tailscale achieves ~94% direct connection rates.

8. Summary

The NAT problem in one sentence: NAT only allows replies to connections that were started from the inside — so if both devices are behind NAT, neither can start a connection to the other.

The solutions all share the same principle: use a publicly reachable server that both devices can connect to (since outgoing connections always work through NAT). They differ in what happens next:

Approach Server Role Data Path
Traditional VPN Permanent relay for all traffic Device → Server → Device
Overlay / P2P mesh Matchmaker only (introduces peers) Device → Device (direct)
Overlay fallback Relay only when direct path fails Device → Relay → Device

For the specific case of an iPhone on 5G communicating with a MacBook on LAN, overlay networks like ZeroTier or Tailscale are the most practical solution. They handle NAT traversal, CGNAT, IP address changes during mobile handovers, and encryption — all transparently, with no server infrastructure to maintain.
